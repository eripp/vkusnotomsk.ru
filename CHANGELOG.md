Все значимые изменения проекта **Вкусно · vkusnotomsk.ru** документируются в этом файле.

Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/).

## [Unreleased]

### 2026-07-01 — Печать листа заказа

#### Added
- **Лист заказа для печати** `/admin/orders/{id}/print` — отдельная чистая
  страница (без меню) с авто-вызовом печати. Для сборки и доставки: № заказа,
  дата/слот, адрес, телефон, клиент, состав (позиции с чек-боксом ✓, кол-во,
  цена, сумма), итоги, способ оплаты + сдача/терминал-плашка. Кнопка «🖨 Печать»
  на странице деталей заказа.

### 2026-07-01 — Роли админки (admin/operator) + раздел «Ключи API»

#### Added
- **Роль `AdminUser.role`** (миграция `0009`): `admin` (полный доступ) и
  `operator` (всё, кроме API-ключей). Учётка оператора из .env
  (`OPERATOR_USERNAME`/`OPERATOR_PASSWORD`), засевается на старте.
- **Раздел «Ключи API»** `/admin/api-settings` — все секретные ключи интеграций
  (ЮKassa, i-dgtl, SmartCaptcha, Plusofon, SMS.RU, Dadata, Яндекс.Карты).
  Закрыт `require_role_admin`: оператору → 404, пункт меню скрыт.

#### Changed
- API-ключи вынесены со страницы «Настройки» в отдельный раздел «Ключи API».
  Общие настройки (оформление, контакты, тексты, метрика, кешбэк) остались
  на `/admin/settings` и доступны оператору.
- Роль пробрасывается в шаблоны админки (`role` в `_tmpl`).

### 2026-06-29 — Аудит действий + rate-limit по IP

#### Added
- **Журнал важных действий** (`audit_events`, миграция `0008`): вход, отправка/
  проверка кода, вход в админку — IP + действие + результат (ok/fail/blocked) +
  детали + время. Хранение 30 дней (автоочистка). Не пишем каждый HTTP-запрос —
  чтобы при нагрузке/DDoS не топить БД.
- **Страница `/admin/audit`**: лента событий с фильтрами (IP/действие/результат)
  и блок «Топ IP за 24 часа». Пункт «Аудит» в меню админки.
- **Rate-limit по IP** (in-memory, `app/services/audit.py`): send-otp 10/5мин,
  verify-otp 20/5мин, admin-login 10/5мин → 429 при превышении.

#### Notes
- Аудит уровня приложения — для разбора постфактум, НЕ защита от DDoS.
  Реальная защита от объёмного флуда — на уровне nginx/Cloudflare/провайдера.

### 2026-06-29 — SMS-вход через Plusofon, детали заказа, возврат TG

#### Added
- **SMS-канал через Plusofon** (`app/services/plusofon.py`): мы генерим код,
  Plusofon доставляет (`POST /api/v1/sms`, header `Client: 10553` + Bearer,
  `number_id` номера-отправителя). Успех по `success: true`. Кнопка SMS активна.
  Креды — в админ-карточке «Plusofon» (DB-override) + конфиг
  `PLUSOFON_CLIENT_ID/_TOKEN/_NUMBER_ID`. SMS.RU понижен до резерва.
- **Детальный просмотр заказа** в `/account → Заказы`: клик раскрывает таблицу
  позиций (цена/шт × кол-во = сумма) и полную раскладку (товары, доставка,
  скидка, списанный/начисленный кешбэк, итого). API отдаёт нужные поля.

#### Changed
- Канал `sms` в `send_otp_code` → Plusofon (если настроен), иначе SMS.RU.
- Вход через **Telegram** возвращён (прямой TG Gateway); Telegram — канал
  по умолчанию, SMS — второй активный; MAX/VK — заглушки.
- `/account`: убран аватар-кружок и блок чипсов позиций (дублировал детали).

### 2026-06-27 — Серверная корзина для залогиненных + чистки входа

#### Added
- **Серверная корзина** (`cart_items`, миграция `0007`): хранит `user_id+product_id+qty`;
  имя/цена/фото берутся из `Product` при чтении (без устаревших снапшотов).
- API `GET /api/cart/items` (подтянуть серверную) и `PUT /api/cart/items`
  (заменить целиком — дебаунс-синк/мердж); оба под `get_optional_user`.
- `cart.js`: у залогиненного каждое изменение пушится на сервер (debounce 500мс).
  Мердж на входе — локальная корзина побеждает; пустая локальная → подтягиваем
  серверную. На выходе корзина в браузере очищается.

#### Changed
- Платные OTP-каналы (SMS/Telegram через i-dgtl) временно отключены:
  `ENABLED_PAID_CHANNELS` пуст, все каналы идут по dev-схеме (код на экране).
  В форме входа активна только заглушка MAX, SMS/Telegram/VK — неактивны.
- `/account`: убраны эмодзи из сайдбара, удалена кнопка «Привязать Telegram».

#### Fixed
- `acctAuth.verifyOtp` читал тело ответа дважды (`res.json()` × 2) — исправлено.

### 2026-06-26 — Защита админки: секретный URL + логин

### 2026-06-26 — Подтверждение входа через i-dgtl (SMS / Telegram Gateway)

#### Added
- **Интеграция i-dgtl** (`app/services/idgtl.py`): провайдер сам генерирует, шлёт
  и проверяет код. `idgtl_send(phone, channel)→uuid`, `idgtl_check(uuid, code)→status`
  (CONFIRMED/WRONG_CODE/EXPIRED/NOT_FOUND). Каналы `sms`→SMS, `tg`→TELEGRAM_GATEWAY.
- **Колонка `otp_codes.provider_uuid`** (миграция `0006`) — храним uuid сессии
  верификации провайдера; для таких кодов `code_hash` пустой.
- **Админ-карточка i-dgtl**: `idgtl_api_key` + `idgtl_gateway_id` (DB-override).
- Конфиг `IDGTL_API_KEY`, `IDGTL_GATEWAY_ID`.
- В форме входа канал **Telegram** снова активен (доставка через i-dgtl).

#### Changed
- `send-otp`/`verify-otp` — гибрид: при настроенном i-dgtl каналы SMS/Telegram идут
  через провайдера (свой код не генерируем); иначе — запасная схема (SMS.RU / показ
  кода на экране). MAX/VK остаются неактивными заглушками.

### 2026-06-26 — Выбор канала подтверждения входа (Telegram / MAX / VK)

#### Added
- **Форма выбора канала** доставки кода на экране входа (модалка `base.html`
  и auth-wall `account.html`): три кнопки с SVG-логотипами — Telegram, MAX, VK;
  Telegram выбран по умолчанию. Стили `.auth-channels` / `.auth-channel`.
- Значение `vk` в enum `OtpChannel` + миграция `0005_otp_vk`.

#### Changed
- `send_otp_code` теперь возвращает `bool` (доставлен ли код). `tg` → Telegram
  Gateway (рабочий), `sms` → SMS.RU, `max`/`vk` → заглушки (код не отправляется).
- `send-otp` показывает `dev_code` на экране, если доставка не состоялась
  (заглушка MAX/VK или канал без баланса/настройки) — раньше решалось эвристикой
  по наличию ключей в `.env`. Канал по умолчанию — `tg`.

#### Notes
- MAX и VK — пока заглушки: мессенджеры не умеют слать «холодный» код по номеру
  (нужна привязка бота). Реальная доставка по номеру — Telegram Gateway или SMS.RU,
  оба требуют пополненного баланса/настроенного отправителя.

### 2026-06-25 — Вход по SMS-коду (SMS.RU), флаг подтверждения телефона

#### Added
- **SMS-канал OTP через SMS.RU** (`send_sms_ru` в `app/services/notifications.py`):
  реальный контракт `GET https://sms.ru/sms/send` (`api_id`, `to` без «+», `msg`,
  `json=1`, опционально `test=1`), разбор верхнеуровневого и по-номерного
  `status_code`. Новый член enum `OtpChannel.sms`. Это единственный канал,
  доставляющий код по номеру телефона.
- **Карточка «SMS.RU» в админ/настройки**: `smsru_api_id` (DB-override над `.env`)
  и чекбокс `smsru_test` (тестовый режим без отправки/списания).
- **Колонки `users.phone_verified` / `phone_verified_via`** + миграция
  `0004_phone_verified` (плюс значение `sms` в enum `otpchannel`). На успешном
  `verify-otp` телефон помечается подтверждённым с указанием канала.
- **Антифлуд на `send-otp`**: серверный cooldown 60с на повторную выдачу кода
  одному номеру (HTTP 429).
- Конфиг `SMSRU_API_ID`.

#### Changed
- Канал по умолчанию для входа — `sms` (модалка и кабинет ранее слали `tg`/`max`).
- `send_max` приведён к реальному контракту MAX Bot API
  (`https://platform-api2.max.ru/messages?user_id=…`, заголовок `Authorization`
  без `Bearer`); адресация по `max_user_id`, не по телефону. Уведомления MAX
  теперь шлются только привязанным юзерам (`user.max_user_id`).

#### Notes
- MAX/VK/Telegram-бот **не умеют** слать «холодный» код по номеру (анти-спам
  платформ) — для входа по коду пригоден только SMS-шлюз или Telegram Gateway.

### 2026-06-22 — Зоны доставки: цена и блокировка вне зоны

#### Fixed
- **Бесплатная доставка не срабатывала**: порог `free_delivery_from` нигде не
  применялся. Теперь на чекауте доставка пересчитывается с учётом порога

  (`inZone`), на сервере — `400` («Адрес вне зоны доставки» / «Укажите адрес на
  карте») для способа «доставка» без попадания в активную зону.

#### Changed
- Чекаут: `.co-fields-row` → колонки `1fr 2fr`; плейсхолдеры полей светлее
