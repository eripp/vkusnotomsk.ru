Группы товаров и варианты — «Котлета с пюре / с гречкой»
Принцип В БД — отдельные товары. В UI — одна карточка с пилюлями выбора варианта.
В каталоге Показывается ОДИН товар от группы (default_product_id). Остальные скрыты.
В модале Все варианты группы → пилюли. Клик → fetch данных выбранного варианта (фото, цена, вес меняются).
URL При переключении варианта → history.replaceState('/product/new-slug')
Корзина Добавляется конкретный product_id выбранного варианта
SEO Каждый вариант = свой slug = своя SSR-страница = индексируется
Запрос в каталог: SELECT DISTINCT ON (group_id) * FROM products WHERE is_visible=true AND is_deleted=false ORDER BY group_id, sort_order — один товар от группы. Товары без group_id показываются все.

Оформление заказа

Поля формы (по порядку)

Телефон Маска +7 (___) ___-__-__. Если авторизован — подставляется автоматически.
Адрес Поиск через Dadata API (autocomplete). После выбора — точка на карте Leaflet + определение зоны.
Зона Автоматически после выбора адреса. Показывает: стоимость доставки, порог бесплатной доставки.
День доставки Пикер дат — только доступные дни (из /api/schedule/available-days).
Время доставки Слоты для выбранного дня (/api/schedule/slots?date=YYYY-MM-DD).
Оплата Радиокнопки: Онлайн (ЮКасса) · Терминал · Наличные + поле «Сдача с ___»
Промокод Поле + «Применить» → валидация на сервере → результат инлайн
Кешбэк Баланс кешбэка + чекбокс «Использовать N ₽» (до X% от суммы, настройка cashback_max_pct)

Расписание доставки — слоты и дедлайны
Часовой пояс Все операции в UTC+7 (Томск). Хранить и сравнивать только в UTC+7.
Тип 1 — Дефолтный Привязан к дню недели (1–7). Повторяется каждую неделю. Несколько записей на один день — разрешено.
Тип 2 — Конкретная дата Приоритет выше дефолтного. Перекрывает дефолт для указанной даты полностью.
Слоты Диапазон доставки / интервал = список слотов. Пример: 19:00–21:00 / 1ч = [19–20, 20–21]
Дедлайн deadline_dt = delivery_date − deadline_days_before (days) + deadline_time
Варианты дедлайна В тот же день (0), за день (1), за 2 дня (2), за 3 дня (3)
День без записей = выходной. Не отображается в пикере дат.
Алгоритм: для каждой из ближайших 14 дат: 1) ищем записи по конкретной дате → если есть, используем только их; 2) иначе — дефолт дня недели; 3) для каждой записи проверяем now(UTC+7) ≤ deadline_dt; 4) генерируем слоты из диапазона. Возвращаем даты с флагом has_slots.

Авторизация — OTP по телефону
Метод Только телефон + OTP. Без паролей. Первый вход = авторегистрация.
Каналы OTP Пользователь выбирает: MAX (через MAX Business API) или Telegram-бот
Код 4-значный. Живёт 5 минут. Максимум 3 попытки, затем новый запрос.
Сессия JWT в httpOnly cookie, TTL 30 дней.
Telegram привязка Deeplink → /start в боте → бот получает chat_id → привязывает к аккаунту по токену.
Ввод телефона→ Выбор канала→ OTP отправлен→ Ввод кода→ JWT выдан ✓

Зоны доставки
Хранение GeoJSON-полигоны в PostgreSQL (поле JSON). Каждая зона — замкнутый полигон.
Определение зоны Python-библиотека Shapely: Point(lon, lat).within(Polygon(coords))
Несколько зон Если точка в нескольких — берётся зона с наибольшим priority.
Вне зон Сообщение «Доставка в ваш район пока недоступна».
Параметры зоны Название, цена доставки, мин. сумма для бесплатной доставки, мин. сумма заказа.

Оплата — ЮКасса + наличные + терминал
ЮКасса SDK yookassa. При создании заказа → создаём Payment → редирект на confirmation_url.
Webhook POST /api/payment/callback → проверяем подпись → обновляем payment_status = 'paid' → начисляем кешбэк → уведомляем клиента.
Терминал / наличные Заказ создаётся сразу. payment_status = 'pending'. Оператор подтверждает оплату вручную в админке.
Отмена онлайн Клиент отменяет → API ЮКассы refund → payment_status = 'refunded'.
14
Промокоды и кешбэк

Тип 1 — Скидка %

Скидывает X% с суммы заказа. Настройки: %, период, лимит использований, мин. сумма заказа.

Тип 2 — Кешбэк покупателю

После доставки начисляется X% от суммы заказа на кешбэк-счёт покупателя.

Тип 3 — Реферальный

Покупатель получает X% кешбэка. Владелец кода (блогер) получает Y% с каждой покупки по его коду.

Трата кешбэка До cashback_max_pct% от суммы заказа (глобальная настройка в site_settings).
Начисление Только после статуса «Доставлен». Не при создании заказа.
Вывод (блогер) Оператор вручную уменьшает баланс в админке → тип транзакции «withdraw» с комментарием.
История Полный лог cashback_transactions — виден пользователю в ЛК и оператору в админке.

Email SMTP. Шаблоны Jinja2 (HTML-письма). Поле users.email — указывается в профиле ЛК.
